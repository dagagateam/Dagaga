# 2026-01-15


## 1) Jenkins–GitLab CI/CD 파이프라인을 쓰는 이유

### Git push를 “배포까지 이어지는 자동 트리거”로 만들기

* Jenkins는 Git 같은 VCS와 연동해서 **커밋(변경)을 감지하면 자동으로 빌드(자동화 테스트 포함)를 돌릴 수** 있게 설정할 수 있다고 설명합니다. 즉, “push → 자동 빌드/테스트”의 출발점이 됩니다.
* 글에서는 GitLab 쪽에 **Webhook을 걸고**, Jenkins에서는 “GitLab에 push되면 빌드 유발” 옵션을 켜서 **GitLab 이벤트가 Jenkins 잡을 실행**하게 만듭니다.

### 배포 서버(EC2)까지 “일관된 절차”로 연결하기

* Jenkins에서 빌드로 생성된 **.jar를 scp로 EC2(배포 서버)에 전송**하고, 이어서 **EC2에서 deploy.sh를 실행**하는 단계까지 연결합니다. 즉 CI(빌드/테스트)에서 CD(배포/실행)로 자연스럽게 이어집니다.

---

## 2) 이 조합의 강점

### (1) 이벤트 기반 자동화: 사람 손을 줄이고 실수를 줄임

* “push 이벤트 → Jenkins 자동 실행(Webhook)”으로 수동 배포/수동 빌드 단계를 줄일 수 있습니다.

### (2) Jenkins의 확장성: 플러그인으로 GitLab/SSH 등 필요한 기능을 붙임

* GitLab 플러그인 설치 + Credentials 설정으로 GitLab 연동을 구성하고
* SSH 플러그인 + SSH Credentials로 EC2 원격 접속/명령 실행을 구성합니다.

### (3) 빌드 도구 표준화: Gradle로 빌드 과정이 명확해짐

* Jenkins Global Tool Configuration에 **Gradle(예: 7.6)을 등록**하고, 잡에서 Gradle build step을 통해 빌드를 수행합니다.

### (4) 운영 환경에 맞춘 현실적인 배포: 아티팩트(jar) 전달 + 서버 스크립트 실행

* 빌드 산출물을 서버로 “복사(scp)”하고 서버에서 “배포 스크립트 실행”하는 방식이라, 운영 서버의 실행 방식(서비스 재시작, 심볼릭 링크 교체 등)을 deploy.sh로 흡수할 수 있습니다.

## 3) 파이프라인 실행 전 테스트/사전 검증 체크리스트

글 자체가 “연동이 되면 다음 단계로 넘어가는” 구조라서, 파이프라인을 돌리기 전에 아래를 **단계별로 미리 확인**하는 게 핵심입니다.

### A. 트리거(연동) 사전 검증: “push가 Jenkins를 깨우는지”

1. **GitLab Access Token 발급 및 안전 보관**(한 번만 보여서 미리 저장)
2. Jenkins에 **GitLab 플러그인 설치 → GitLab API token Credentials 등록 → 시스템 설정에 GitLab Connection 등록**
3. Jenkins 잡에서 **“Build when a change is pushed to GitLab” + Webhook URL/Secret token 생성**
4. GitLab 프로젝트 Webhooks에 **Jenkins Webhook URL/Secret token 설정** 후 **Test → Push events로 테스트**

👉 여기까지 통과하면 “코드 변경 이벤트가 CI 파이프라인을 트리거”한다는 가장 중요한 연결이 확보됩니다.

### B. 빌드/테스트 사전 검증: “빌드가 재현 가능하게 도는지”

* Jenkins에 **Gradle 등록** 후 잡의 Build Step에서 Gradle 빌드가 정상 동작하는지 확인
* (글에서 Jenkins는 “자동화 테스트 포함 빌드”가 가능하다고 설명) 따라서 Gradle 단계에서 일반적으로는 `test`가 함께 실행되도록 구성하는 게 CI 품질의 핵심입니다.

### C. 배포(서버) 사전 검증: “전송 + 원격 실행이 되는지”

* Jenkins에서 EC2로 갈 **SSH Credential(개인키)** 등록 및 SSH Remote hosts 설정(Hostname/Port/Creds)
* Jenkins 컨테이너에서 **scp로 jar 전송**이 되는지(권한/경로/키/StrictHostKeyChecking 옵션 포함)
* **deploy.sh 실행 경로/권한**이 맞는지(잡에서 SSH로 해당 스크립트 실행)
* 참고로 글에는 “EC2 Ubuntu 22.04는 Jenkins에서 연결이 안 됐다”는 경험이 있어서, OS/SSH 호환성도 사전 체크 포인트로 언급됩니다.

---

## 4) 한 줄 요약

**GitLab Push → Webhook → Jenkins Job 실행 → Gradle 빌드(테스트 포함 가능) → 산출물(jar) scp로 EC2 전송 → EC2에서 deploy.sh 실행**

원하면, 위 내용을 바탕으로 **(1) Jenkins Freestyle 기준 단계별 “권장 Job 구성(빌드/테스트/배포 단계 분리)”** 또는 **(2) 같은 구성을 Jenkinsfile(선언형 파이프라인)로 바꾼 예시 구조**까지도 깔끔하게 정리해줄게요.
