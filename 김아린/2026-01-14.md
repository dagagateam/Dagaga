# 1. 문제 배경

당근마켓의 검색 트래픽은 “지역”에 매우 강하게 종속된다.

* 사용자는 대부분 자기 동네 반경 몇 km 안의 글만 검색
* 특정 인기 지역(강남, 판교, 해운대 등)에 글과 트래픽이 집중
* 기존 구조: 모든 샤드에 쿼리를 브로드캐스트 → 결과를 합쳐 랭킹

이 구조의 문제:

* 필요 없는 샤드까지 전부 검색
* 인기 지역 하나의 폭증이 전체 검색 클러스터 성능에 영향을 줌
* 인프라 비용이 계속 증가

Hot/Cold(최근 글 / 오래된 글 분리) 구조도 시도했지만,

* 랭킹이 끊기고
* 지역 트래픽 격리가 되지 않아 근본 해결이 되지 않았다.

---

# 2. 핵심 아이디어: “지역 기반 샤딩”

검색 요청이 특정 지역에 대해서만 의미가 있으므로:

> “이 지역과 관련된 샤드만 검색하게 만들자”

이를 위해:

* 모든 지역을 몇 개의 샤드 그룹으로 나누고
* 요청 지역이 속한 샤드 그룹만 검색하도록 라우팅

---

# 3. 전체 아키텍처

구성 요소:

```
[Client]
   |
   v
[Search API]
   |
   v
[Shard Router]
   |
   +--> [Shard A (지역 묶음 1)]
   +--> [Shard B (지역 묶음 2)]
   +--> [Shard C (지역 묶음 3)]
              ...
```

동작 흐름:

1. 클라이언트가 검색 요청 (키워드 + 지역ID)을 보냄
2. Search API가 Shard Router에 지역ID를 전달
3. Shard Router가 해당 지역이 속한 shard_group_id를 조회
4. 해당 shard_group에만 쿼리를 전달
5. 결과를 받아서 랭킹 후 응답

# 4. 샤드 정보 테이블 설계

지역 → 샤드 매핑을 위한 테이블이 핵심이다.

## 4.1 기본 테이블

```
region_shard_mapping
--------------------------------
region_id        bigint (PK)
shard_group_id   int
region_level     int   -- 동, 구, 시 같은 행정 레벨
parent_region_id bigint
document_count   bigint
updated_at       timestamp
```

* region_id: 동/읍/구 단위 지역 ID
* shard_group_id: 이 지역이 속한 샤드 그룹
* document_count: 해당 지역의 게시글 수
* parent_region_id: 상위 행정구역 (구 → 시 → 도)

이 테이블은:

* 배치 작업으로 주기적으로 재계산
* Shard Router 서버 메모리에 로드됨

## 4.2 샤드 그룹 테이블

```
shard_group
--------------------------
shard_group_id   int (PK)
primary_node     string
replica_nodes    json
total_docs       bigint
```

* 각 샤드 그룹이 어느 검색 노드들로 구성되는지 정의

# 5. 클러스터링: Weighted K-Means

단순히 지역을 나누면 문제가 생긴다.

* 서울 강남 = 게시글 100만
* 시골 한 동네 = 게시글 1천

이걸 동일 취급하면 한 샤드가 터진다.

그래서 **가중치 기반 K-Means**를 사용한다.

## 5.1 입력 데이터

각 지역은 다음 벡터로 표현된다.

```
Region i = (x_i, y_i, w_i)

x_i, y_i : 지역의 위도/경도
w_i      : document_count (게시글 수)
```

## 5.2 목표

* K개의 샤드 그룹을 만들되
* 각 샤드의 총 document_count 합이 비슷
* 그리고 지리적으로 인접한 지역끼리 묶이도록

## 5.3 Weighted K-Means 개념

일반 K-Means는:

```
minimize Σ ||point - centroid||²
```

Weighted K-Means는:

```
minimize Σ w_i * ||point_i - centroid||²
```

즉,

* 게시글이 많은 지역은 더 큰 영향력을 가짐
* 트래픽이 많은 지역이 샤드 균형에 더 크게 반영됨

## 5.4 결과

* 강남, 판교, 잠실 같은 밀집 지역은 서로 다른 샤드로 분산
* 시골 지역들은 여러 개가 하나의 샤드로 묶임
* 각 샤드의 총 문서 수가 거의 비슷해짐

# 6. 코드 구조 (개념적)

## 6.1 Shard Router
참고: 이 코드 예시는 kotlin으로 작성되어있음

```kotlin
class ShardRouter(
    val regionShardMap: Map<Long, Int> // region_id -> shard_group_id
) {
    fun route(regionId: Long): Int {
        return regionShardMap[regionId] ?: DEFAULT_SHARD
    }
}
```

## 6.2 Search API

```kotlin
fun search(keyword: String, regionId: Long): SearchResult {
    val shardGroup = shardRouter.route(regionId)
    val nodes = shardGroupRepository.getNodes(shardGroup)

    val responses = nodes.parallelMap { node ->
        node.search(keyword, regionId)
    }

    return mergeAndRank(responses)
}
```

핵심:

* 모든 샤드에 보내지 않고
* 특정 shard_group에만 쿼리를 보냄

# 7. 효과

이 구조 도입 후:

* 평균 검색 응답 시간 약 50% 이상 감소
* 클라우드 비용 약 53% 절감
* 특정 인기 지역 폭주가 다른 지역 검색에 영향을 주지 않음
* CPU 사용률 편차 감소

---

# 8. 운영상 주의점

Shard Router는 단일 장애점(SPOF)이 될 수 있다.

그래서:

* 다중 인스턴스
* 샤드 매핑 캐시를 로컬 메모리에 유지
* 장애 시 기본 샤드로 폴백

이런 안정화 설계가 필수다.
